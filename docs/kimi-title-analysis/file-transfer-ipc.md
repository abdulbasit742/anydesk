# Secure File Transfer IPC Design

This document details the Inter-Process Communication (IPC) design for secure file transfer within the RemoteDesk Electron application. The primary goal is to ensure that file operations are performed safely, preventing unauthorized access, path traversal vulnerabilities, and other common security risks.

## Core Security Principles

1.  **Renderer Process Isolation**: The renderer process, which executes untrusted web content, is strictly isolated from direct file system access. All file operations are mediated by the Electron main process.
2.  **Main Process Mediation**: The Electron main process acts as a trusted intermediary, performing all file system interactions on behalf of the renderer. It enforces security policies and sanitizes all inputs.
3.  **Token-Based Access Control**: File handles and paths are not directly exposed to the renderer. Instead, a unique, short-lived token is generated by the main process for each authorized file operation. The renderer must present this token for subsequent interactions related to that specific file.
4.  **No Arbitrary Path Writes**: The renderer cannot specify arbitrary paths for writing files. Save locations are chosen by the user via a native system dialog, and the main process ensures writes occur only within the explicitly approved path.
5.  **Filename Sanitization**: All incoming filenames are rigorously sanitized to prevent path traversal attacks (e.g., `../`, `..\`), invalid characters, and the use of Windows reserved names.
6.  **Size Limits**: File transfers are subject to maximum size limits to prevent resource exhaustion and potential denial-of-service attacks.

## IPC Channels and Flow

### `fileTransfer:pickFile` (Renderer -> Main)

*   **Purpose**: Allows the renderer to initiate a file selection dialog for sending a file.
*   **Main Process Action**: Displays a native `openFile` dialog. If a file is selected and passes initial size checks, the main process generates a unique token, stores the file's metadata and path associated with this token, and returns the metadata (including the token) to the renderer.
*   **Security**: The renderer never sees the actual file path, only the metadata and the token.

### `fileTransfer:chooseSaveLocation` (Renderer -> Main)

*   **Purpose**: Allows the renderer to initiate a save dialog for an incoming file.
*   **Main Process Action**: Displays a native `saveFile` dialog, pre-filling the filename with a sanitized version of the suggested name. If a location is chosen, the main process generates a unique token, stores the chosen save path associated with this token, and returns the path and token to the renderer.
*   **Security**: The renderer cannot dictate the save path; the user makes the final decision. The filename is sanitized before being presented to the user and used for saving.

### `fileTransfer:readFileChunk` (Renderer -> Main)

*   **Purpose**: Requests a specific chunk of a file to be read for sending.
*   **Main Process Action**: Validates the provided token against active read handles. If valid, it reads the specified chunk from the associated file path and returns it (Base64 encoded) to the renderer.
*   **Security**: Ensures that the renderer can only read from files it was explicitly granted access to via a token. No direct path access.

### `fileTransfer:writeFileChunk` (Renderer -> Main)

*   **Purpose**: Sends a chunk of data to be written to an incoming file.
*   **Main Process Action**: Validates the provided token against active write handles. If valid, it writes the Base64 decoded chunk data to the associated file path at the specified offset.
*   **Security**: Prevents arbitrary writes. Data is written only to the file path associated with the token, which was user-approved. The main process manages the file descriptor.

### `fileTransfer:cancelTransfer` (Renderer -> Main)

*   **Purpose**: Notifies the main process to cancel an ongoing transfer and clean up resources.
*   **Main Process Action**: Closes the file handle associated with the token and removes it from the active handles map.

## File Handle Token Model

Instead of passing raw file paths, the main process generates a `token` (UUID) for each file operation that requires ongoing access (e.g., reading chunks from a selected file, writing chunks to a chosen save location). This token is then used by the renderer to refer to that specific file operation.

```typescript
// Main process logic (simplified)
const activeFileHandles = new Map<string, { filePath: string; token: string; isWriting: boolean; fileDescriptor?: fs.FileHandle; }>;

ipcMain.handle("fileTransfer:pickFile", async () => {
  // ... user picks file, get actualPath
  const token = uuidv4();
  activeFileHandles.set(token, { filePath: actualPath, token, isWriting: false });
  return { metadata, token };
});

ipcMain.handle("fileTransfer:readFileChunk", async (_event, token: string, offset: number, length: number) => {
  const handle = activeFileHandles.get(token);
  if (!handle || handle.isWriting) { /* error */ }
  // ... read from handle.filePath
});
```

This token-based approach ensures that the renderer never directly holds or manipulates sensitive file system paths, significantly reducing the attack surface.

## Sanitization and Validation

All filenames, especially for incoming files, are passed through `sanitizeFilename` (from `packages/shared/src/fileTransfer/filenameSanitizer.ts`) to neutralize malicious inputs. This includes:

*   Removing path traversal sequences (`../`, `..\`).
*   Filtering out invalid characters.
*   Preventing the use of Windows reserved device names (e.g., `CON`, `PRN`).
*   Enforcing maximum filename length.

File sizes are also validated against `MAX_FILE_SIZE_BYTES` in the main process before any transfer begins, preventing attempts to transfer excessively large files.
