# RemoteDesk Desktop Client: WebRTC Peer Creation Test Plan

This test plan focuses on verifying the successful creation and initial setup of WebRTC peer connections within the RemoteDesk desktop client. This is a foundational step for establishing remote desktop sessions.

## 1. Test Objectives

*   Verify that `RTCPeerConnection` objects are correctly instantiated.
*   Verify that ICE candidates are generated and exchanged.
*   Verify that local and remote descriptions (offer/answer) are set successfully.
*   Verify that data channels are created and ready for use.

## 2. Test Environment

*   RemoteDesk Desktop Client running (both host and viewer instances, ideally on separate machines or VMs for full WebRTC testing).
*   Signaling server (API service) running and accessible.
*   TURN server (if used) running and accessible.

## 3. Test Cases

### 3.1 TC-WPC-001: Successful Peer Connection Establishment (Host)

*   **Description**: Verify that the host client can successfully create an `RTCPeerConnection`, generate an offer, and set its local description.
*   **Steps**:
    1.  Launch the RemoteDesk desktop client as a host.
    2.  Initiate a connection request to a viewer (or simulate this action).
    3.  Observe the client's internal logs (renderer DevTools console, main process logs).
*   **Expected Result**:
    *   An `RTCPeerConnection` object is instantiated.
    *   A WebRTC offer (SDP) is generated.
    *   The local description is successfully set on the `RTCPeerConnection`.
    *   ICE candidates start being gathered.

### 3.2 TC-WPC-002: Successful Peer Connection Establishment (Viewer)

*   **Description**: Verify that the viewer client can successfully create an `RTCPeerConnection`, set a remote offer, generate an answer, and set its local description.
*   **Steps**:
    1.  Launch the RemoteDesk desktop client as a viewer.
    2.  Receive a connection offer from a host (or simulate this action).
    3.  Observe the client's internal logs.
*   **Expected Result**:
    *   An `RTCPeerConnection` object is instantiated.
    *   The remote description (offer) is successfully set.
    *   A WebRTC answer (SDP) is generated.
    *   The local description is successfully set on the `RTCPeerConnection`.
    *   ICE candidates start being gathered.

### 3.3 TC-WPC-003: ICE Candidate Exchange

*   **Description**: Verify that ICE candidates are successfully generated by both peers and exchanged via the signaling server.
*   **Steps**:
    1.  Establish a peer connection between a host and a viewer.
    2.  Observe the network traffic and client logs for `onicecandidate` events.
    3.  Verify that `webrtc:ice` messages are sent to and received from the signaling server.
*   **Expected Result**:
    *   Both peers generate ICE candidates.
    *   ICE candidates are sent to the signaling server.
    *   ICE candidates received from the signaling server are successfully added to the `RTCPeerConnection`.
    *   The `iceconnectionstate` eventually transitions to `connected` or `completed`.

### 3.4 TC-WPC-004: Data Channel Creation

*   **Description**: Verify that the necessary `RTCDataChannel`s (e.g., for `SessionDataChannel`, clipboard, file transfer) are created and open successfully.
*   **Steps**:
    1.  Establish a peer connection between a host and a viewer.
    2.  Observe client logs for `ondatachannel` events (on the receiving side) and `dataChannel.onopen` events.
*   **Expected Result**:
    *   The host successfully creates data channels.
    *   The viewer successfully receives and opens the data channels.
    *   Data channels report their `readyState` as `open`.

### 3.5 TC-WPC-005: TURN Server Usage (if configured)

*   **Description**: Verify that the client correctly utilizes the configured TURN server for relaying if direct peer-to-peer connection fails.
*   **Steps**:
    1.  Configure clients to use a TURN server.
    2.  Simulate a network environment where direct P2P connection is blocked (e.g., strict NAT).
    3.  Establish a peer connection.
    4.  Observe ICE candidate types in client logs (`relay` candidates).
*   **Expected Result**: The connection is established via the TURN server, and `relay` candidates are observed.

### 3.6 TC-WPC-006: Error Handling During Peer Creation

*   **Description**: Verify that the client handles errors gracefully during `RTCPeerConnection` setup.
*   **Steps**:
    1.  Attempt to create a peer connection with invalid SDP or ICE candidates.
    2.  Simulate signaling server unavailability.
*   **Expected Result**: The client logs errors, attempts re-connection if appropriate, and provides user feedback without crashing.
