# RemoteDesk Two-Client Local Test Checklist

This checklist provides a structured approach to verifying the core functionalities of RemoteDesk using a two-client local test setup (Host and Viewer).

## Setup Verification

*   [ ] Both Host and Viewer desktop clients are running.
*   [ ] Both clients are logged in with distinct user accounts.
*   [ ] The API service (including Socket.IO signaling) is running and accessible.
*   [ ] The database is running and accessible.
*   [ ] (Optional) Local TURN server is running and configured if testing relay scenarios.

## Connection Flow

*   [ ] Host client displays its RemoteDesk ID correctly.
*   [ ] Viewer client can successfully enter Host's RemoteDesk ID.
*   [ ] Viewer client can initiate a connection request.
*   [ ] Host client receives and displays the incoming connection request.
*   [ ] Host client can accept the connection request.
*   [ ] Host client can reject the connection request.
*   [ ] Viewer client receives appropriate feedback (connection accepted/rejected).

## Screen Streaming

*   [ ] After acceptance, Host client presents the screen source picker.
*   [ ] Host client can select a screen/window to share.
*   [ ] Viewer client displays the Host's screen stream.
*   [ ] Stream quality is acceptable (resolution, frame rate).
*   [ ] Latency is minimal for interactive use.
*   [ ] Dynamic content on Host's screen is reflected on Viewer's screen.
*   [ ] Host can stop sharing the screen.
*   [ ] Viewer client correctly handles stream termination.

## WebRTC Functionality

*   [ ] ICE candidates are generated by both peers.
*   [ ] ICE candidates are exchanged via the signaling server.
*   [ ] `RTCPeerConnection` state transitions to `connected` or `completed`.
*   [ ] (If TURN is configured) `relay` candidates are observed in `chrome://webrtc-internals` when P2P fails.
*   [ ] Data channels are established and open.

## Data Channel Features (Clipboard Sync & File Transfer)

*   [ ] **Clipboard Sync**:
    *   [ ] Both Host and Viewer enable clipboard sync.
    *   [ ] Text copied on Host appears on Viewer's clipboard.
    *   [ ] Text copied on Viewer appears on Host's clipboard.
    *   [ ] Disabling sync on one side stops sync for both.
    *   [ ] Large text content (within limits) syncs correctly.
*   [ ] **File Transfer**:
    *   [ ] Host can initiate a file transfer.
    *   [ ] Viewer receives a consent dialog with correct file info.
    *   [ ] Viewer can accept the file transfer.
    *   [ ] Viewer can reject the file transfer.
    *   [ ] File transfers successfully upon acceptance.
    *   [ ] Transferred file integrity is maintained.
    *   [ ] Progress bar, speed, and ETA are displayed correctly.
    *   [ ] Both Host and Viewer can cancel an ongoing transfer.
    *   [ ] Filename sanitization prevents path traversal (test with malicious names).

## Remote Input (if enabled)

*   [ ] Host can enable/disable remote input permissions.
*   [ ] Viewer can send mouse/keyboard input to Host.
*   [ ] Host receives remote input (native execution intentionally disabled for security).
*   [ ] Emergency stop functionality works as expected.

## Disconnection and Reconnection

*   [ ] Host can gracefully end the session.
*   [ ] Viewer can gracefully end the session.
*   [ ] Both clients handle abrupt disconnections (e.g., closing the app) without crashing.
*   [ ] Reconnecting after a session ends works correctly.

## Logging and Debugging

*   [ ] Relevant logs are generated in both client DevTools and main process.
*   [ ] API service logs show signaling events.
*   [ ] `chrome://webrtc-internals` provides useful debugging information.
