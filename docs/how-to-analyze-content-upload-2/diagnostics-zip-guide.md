# Diagnostics Zip Guide for RemoteDesk

This document outlines the process and best practices for collecting and zipping diagnostic information for RemoteDesk.

## Overview
When users encounter issues with RemoteDesk, collecting comprehensive diagnostic information is crucial for effective troubleshooting. This guide details how to gather relevant system, network, session, and application logs, and package them into a single, compressed archive for easy submission to support.

## Components to Collect
The diagnostic zip file should ideally include:
- **System Information**: OS version, CPU, memory, uptime, etc.
- **Network Information**: IP configuration, DNS servers, default gateway, network interfaces.
- **WebRTC Statistics**: Detailed statistics from active WebRTC peer connections (e.g., packet loss, jitter, bandwidth).
- **Session Logs**: Application-specific logs for the RemoteDesk session, including signaling, WebRTC events, and error messages.
- **Crash Dumps (if available)**: Relevant crash reports generated by the operating system or application.
- **Configuration Files**: Anonymized application configuration settings.

## Process for Generating a Diagnostic Zip

### 1. Triggering Diagnostics Collection
- **In-App Button**: Provide a clear "Collect Diagnostics" or "Report Issue" button within the RemoteDesk application UI.
- **Command Line (Advanced Users)**: Offer a command-line option for headless or advanced users to trigger diagnostic collection.

### 2. Data Collection
- The application will programmatically gather information using internal APIs and system commands (as outlined in `system-info-collector.ts`, `network-info-collector.ts`, `webrtc-stats-exporter.ts`, `session-log-exporter.ts`).
- **Asynchronous Collection**: Data collection should be asynchronous to avoid freezing the UI.

### 3. Privacy Redaction
- **Sensitive Information**: Before zipping, sensitive information (e.g., IP addresses, personal data, API keys) must be redacted or anonymized. Refer to `privacy-redaction-docs.md`.
- **User Consent**: Inform the user about what data is being collected and offer an option to review or further redact before submission.

### 4. Zipping the Data
- All collected diagnostic files (JSON reports, log files, etc.) should be placed into a temporary directory.
- This directory is then compressed into a single `.zip` archive.

#### Example (Conceptual Node.js/Electron)
```typescript
import { collectDiagnostics } from "./diagnostics-collector";
import { promises as fs } from "fs";
import path from "path";
import archiver from "archiver"; // Requires `npm install archiver`

export async function zipDiagnostics(outputPath: string, redact: boolean = true): Promise<string> {
  const diagnosticReport = await collectDiagnostics(redact);
  const timestamp = new Date().toISOString().replace(/[:.]/g, "-");
  const tempDir = path.join(os.tmpdir(), `remotedesk-diagnostics-${timestamp}`);
  await fs.mkdir(tempDir, { recursive: true });

  const reportFilePath = path.join(tempDir, `diagnostic-report-${timestamp}.json`);
  await fs.writeFile(reportFilePath, JSON.stringify(diagnosticReport, null, 2));

  // In a real app, you might also copy actual log files, crash dumps, etc.
  // For example:
  // const sessionLogPath = path.join(getLogDirectory(), `session-${diagnosticReport.sessionId}.log`);
  // if (await fs.access(sessionLogPath).then(() => true).catch(() => false)) {
  //   await fs.copyFile(sessionLogPath, path.join(tempDir, `session-log-${timestamp}.log`));
  // }

  const archive = archiver("zip", { zlib: { level: 9 } });
  const output = fs.createWriteStream(outputPath);

  return new Promise((resolve, reject) => {
    output.on("close", () => {
      console.log(`Diagnostic zip created: ${outputPath} (${archive.pointer()} bytes)`);
      fs.rm(tempDir, { recursive: true, force: true }); // Clean up temp directory
      resolve(outputPath);
    });
    archive.on("error", (err) => reject(err));

    archive.directory(tempDir, false); // Append tempDir contents to root of zip
    archive.finalize();
  });
}
```

### 5. Output and Submission
- The generated zip file should be saved to a user-accessible location (e.g., Desktop, Downloads folder).
- Provide an option to automatically upload the zip file to a secure support portal or attach it to an email.

## Security and Privacy
- **Encryption**: Ensure the diagnostic zip is encrypted if it contains any potentially sensitive (even if redacted) information and is being transmitted over insecure channels.
- **Access Control**: Restrict access to diagnostic files on the server side.
- **Retention Policy**: Define a clear retention policy for diagnostic data.

## Testing
- Verify that all specified components are collected and included in the zip file.
- Test the redaction process to ensure sensitive information is correctly removed.
- Verify the zip file is valid and can be extracted.
- Test on different operating systems and scenarios (e.g., with and without active sessions).
- Ensure the process is robust and handles errors gracefully.
