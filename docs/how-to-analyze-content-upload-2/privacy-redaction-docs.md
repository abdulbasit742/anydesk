# Privacy Redaction Documentation for RemoteDesk Diagnostics

This document outlines the policies and methods for redacting sensitive information from diagnostic reports generated by RemoteDesk.

## Overview
Diagnostic reports are crucial for troubleshooting and improving RemoteDesk. However, they may inadvertently contain sensitive user information, such as IP addresses, usernames, file paths, or parts of chat logs. To protect user privacy and comply with data protection regulations (e.g., GDPR, CCPA), it is essential to redact or anonymize this information before reports are shared with support or development teams.

## Principles of Redaction
- **Minimize Data Collection**: Only collect data that is strictly necessary for diagnosing the issue.
- **Redact by Default**: Sensitive information should be redacted by default, with an option for users to include more detailed (potentially sensitive) information if explicitly requested by support and with clear consent.
- **User Control**: Users should be informed about what data is being collected and have the opportunity to review and approve the redaction process.
- **Irreversibility**: Redaction should be irreversible where possible, ensuring that original sensitive data cannot be reconstructed from the redacted version.

## Types of Information to Redact

### 1. Network Information
- **Public IP Addresses**: Replace with placeholders (e.g., `[REDACTED_IP]`).
- **Private IP Addresses**: Can often be kept if they are internal to a user's network and not publicly routable, but should be considered for redaction if there's a risk of revealing network topology.
- **MAC Addresses**: Replace with placeholders.
- **DNS Server Addresses**: Replace with placeholders if they reveal private network infrastructure.

### 2. User Identifiable Information (PII)
- **Usernames/Email Addresses**: Replace with generic identifiers (e.g., `[USER_ID]`).
- **Hostnames/Device Names**: Replace with generic identifiers (e.g., `[DEVICE_NAME]`).
- **File Paths**: Redact specific user directories (e.g., `/home/user/documents` -> `/home/[USER]/documents`).
- **Chat Messages/Clipboard Content**: If included in logs, these must be heavily redacted or entirely removed.

### 3. Session-Specific Data
- **Session IDs**: While necessary for linking diagnostics, ensure they are not easily linkable to PII without additional internal systems.
- **WebRTC Candidate Addresses**: Redact public IP addresses from ICE candidates.

### 4. System Configuration
- **Sensitive Environment Variables**: Ensure no API keys, tokens, or passwords are inadvertently captured.
- **Registry Keys/Configuration Files**: Redact any values that contain PII or security-sensitive information.

## Redaction Methods

### 1. String Replacement
- Simple find-and-replace for known patterns (e.g., regex for IP addresses, email formats).

### 2. Hashing/Tokenization
- For certain identifiers, one-way hashing or tokenization can be used to maintain uniqueness for analysis while obscuring the original value.

### 3. Data Masking
- Replacing parts of a string with asterisks or other characters (e.g., `192.168.1.100` -> `***.***.1.100`).

## Implementation (`privacy-redaction.ts`)

```typescript
export function redactPrivateInfo(data: string): string {
  let redactedData = data;

  // Redact common patterns for IP addresses
  redactedData = redactedData.replace(/\b(?:\d{1,3}\.){3}\d{1,3}\b/g, '[REDACTED_IP]');

  // Redact email addresses
  redactedData = redactedData.replace(/\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b/g, '[REDACTED_EMAIL]');

  // Redact common user directory paths (Linux/macOS example)
  redactedData = redactedData.replace(/\/home\/[^\/]+\//g, '/home/[USER]/');
  redactedData = redactedData.replace(/\/Users\/[^\/]+\//g, '/Users/[USER]/');

  // Redact Windows user directory paths
  redactedData = redactedData.replace(/C:\\Users\\([^\\]+)\\/g, 'C:\\Users\\[USER]\\\');

  // Redact hostnames (simple example, might need more context-aware redaction)
  redactedData = redactedData.replace(/hostname: ([a-zA-Z0-9\-]+)/g, 'hostname: [REDACTED_HOSTNAME]');

  // Redact specific sensitive keywords or patterns from logs if necessary
  // Example: redactedData = redactedData.replace(/API_KEY=[^\s]+/g, 'API_KEY=[REDACTED]');

  return redactedData;
}
```

## User Interface
- The diagnostic collection UI should clearly state that sensitive information will be redacted.
- Provide a checkbox or option for advanced users to disable redaction (with a strong warning) if full, unredacted logs are specifically requested by support for complex issues.

## Testing
- Thoroughly test the redaction logic with various types of diagnostic data to ensure all sensitive information is correctly identified and redacted.
- Verify that redaction does not inadvertently remove critical diagnostic information needed for troubleshooting.
- Test with different locales and character sets if applicable.
